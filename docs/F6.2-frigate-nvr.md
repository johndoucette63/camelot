# Frigate NVR — Local AI Camera Surveillance

## Overview

Deploy Frigate NVR as a Dockerized service on HOLYGRAIL (Ryzen 7 7800X3D, RTX 2070 Super) to provide local AI-powered video surveillance, event recording, and deep Home Assistant integration. Phase 1 onboards the existing Reolink Video Doorbell WiFi via RTSP; Phase 2 (future) adds Reolink Altas PT Ultra solar cameras through a Reolink Home Hub. The system must operate fully offline — no cloud dependency — and the architecture must support Phase 2 expansion through config additions rather than restructuring.

The primary UX driver is a **Nest-like, wife-friendly event review experience** in the Frigate web UI: glanceable dashboard of recent activity thumbnails, vertical timeline for browsing past events, and filtering by date/camera/object. This UX goal is acceptance-testable and gates release.

See `docs/INFRASTRUCTURE.md` for HOLYGRAIL hardware and Docker topology, and `docs/F6.1-home-assistant-integration.md` for the HA integration this feature builds on.

## User Stories

### US-1: Frigate + go2rtc stack on HOLYGRAIL with GPU passthrough

**As a** Camelot admin
**I want** Frigate running as a Docker container on HOLYGRAIL with go2rtc as an RTSP restreamer and NVIDIA GPU passthrough
**So that** camera streams are consumed once and shared, and AI detection runs on the RTX 2070 Super

#### Acceptance Criteria — US-1

- Frigate defined in docker-compose alongside the existing Ollama stack, image tag pinned
- go2rtc deployed as the RTSP proxy/restreamer for all camera streams (either sidecar or Frigate's built-in go2rtc)
- NVIDIA Container Toolkit configured and verified for GPU passthrough to Frigate (shared with Ollama on the same host)
- Frigate config persisted at `/opt/frigate/config`
- Config structure uses named camera stubs so Phase 2 cameras can be appended without restructuring
- `docker compose up` brings the stack up cleanly on a HOLYGRAIL reboot

### US-2: Reolink Video Doorbell onboarded via go2rtc

**As a** Camelot admin
**I want** the Reolink Video Doorbell WiFi streaming into Frigate via go2rtc (not directly)
**So that** Frigate and the native Reolink HA integration can coexist without triggering the Reolink firmware's multi-connection drop bug

#### Acceptance Criteria — US-2

- Doorbell has a static DHCP reservation on the router (not on-device static IP)
- RTSP and ONVIF enabled on the doorbell via the Reolink app (Settings → Network → Advanced)
- Doorbell password uses only Reolink-safe characters: `a-z A-Z 0-9 @$*~_-+=!?.,:;'()[]`
- go2rtc pulls the doorbell stream; Frigate and the native Reolink HA integration both consume from go2rtc — never from the camera directly
- Stream health verified in go2rtc UI (no disconnects, no frame drops under normal load)

### US-3: GPU-accelerated object detection via TensorRT

**As a** Camelot admin
**I want** Frigate running object detection on the RTX 2070 Super via TensorRT
**So that** detections are fast, accurate, and do not load the CPU

#### Acceptance Criteria — US-3

- Primary detector: NVIDIA GPU (RTX 2070 Super) via TensorRT
- Detection model: YOLOv8 or YOLOv9, version pinned in config
- Sub-stream used for detection (low-res); main stream used for recording
- Tracked objects: `person`, `car`, `dog`, `cat`, `package`
- `nvidia-smi` on HOLYGRAIL shows Frigate as a GPU process during live detection
- Detection latency observed in Frigate logs is acceptable for real-time use (sub-second)

### US-4: Event recording and 24/7 footage retention

**As a** Camelot admin
**I want** continuous recording plus event clips stored on a dedicated drive with tunable retention
**So that** I can review any moment from the last week or any event from the last month

#### Acceptance Criteria — US-4

- Dedicated drive or partition mounted at `/mnt/frigate` for Frigate footage
- 24/7 continuous recording enabled for the doorbell
- Event clips retained 30 days; continuous footage retained 7–14 days (per-camera tunable)
- Events and recordings stored such that HA can reference event clips independently
- Retention values documented in the Frigate config file

### US-5: Home Assistant integration via MQTT

**As a** Camelot admin
**I want** Frigate detections flowing into Home Assistant via MQTT with HACS integration installed
**So that** automations can fire on detections and deliver rich notifications

#### Acceptance Criteria — US-5

- MQTT broker (Mosquitto) running — on HOLYGRAIL or the HA host — and reachable from Frigate
- Frigate HA integration installed via HACS and connected to the MQTT broker
- HA exposes Frigate entities: visitor, person, package sensors per camera
- Automations implemented for:
  - Doorbell press → rich push notification with snapshot + live-stream link
  - Person detected at front door → porch lights on
  - Package detected → notification with clip
  - Privacy mode toggle via HA switch entity
- End-to-end notification test (detection → HA → phone) passes

### US-6: Wife-friendly Frigate web UI for event review

**As a** household member
**I want** a Nest-like dashboard to glance at recent activity and scrub through past events
**So that** reviewing who came to the door is effortless, not technical

#### Acceptance Criteria — US-6

- Frigate web UI reachable on the local network via browser
- Dashboard shows recent activity thumbnails across all cameras
- Vertical timeline view allows browsing past events
- Filter by date, camera, and object type works
- Acceptance test with a non-technical household member: browsing the timeline, viewing thumbnails, and filtering by date are all intuitive without guidance

### US-7: Phase 2 expansion readiness

**As a** Camelot admin
**I want** the docker-compose and Frigate config structured so Phase 2 cameras can be added incrementally
**So that** onboarding the Reolink Home Hub and solar cameras later requires only config additions

#### Acceptance Criteria — US-7

- `docker-compose.yml` and `config.yml` contain commented stubs for Phase 2 cameras under the Home Hub
- Battery/solar-camera RTSP/go2rtc conventions documented (cameras connect through the Hub, not directly)
- A "Phase 2 expansion checklist" is committed in `docs/` describing the end-to-end steps to add a new camera
- The checklist is self-service: a future me could follow it without re-reading the whole spec

## Out of Scope (Phase 1)

- Reolink Home Hub and solar camera hardware/installation — deferred to Phase 2
- Two-way audio via Frigate — upstream support is in-progress; use the Reolink app for now
- Remote access / Tailscale setup — separate feature
- Viewu iOS app setup — nice-to-have; document in the Phase 2 checklist
- Cloud backup of footage — fully local by design

## Risks

- **Reolink multi-connection bug**: pulling streams simultaneously from Frigate and the native HA integration crashes the doorbell firmware. Mitigation: go2rtc relays the single upstream connection (US-2).
- **GPU contention with Ollama**: Frigate and Ollama share the RTX 2070 Super. Detection is lightweight relative to LLM inference, but simultaneous heavy load could degrade both. Monitor with `nvidia-smi` and plan a detector fallback (CPU or Coral) if needed.
- **Storage growth**: 24/7 recording plus event clips fills disk quickly. `/mnt/frigate` must be a dedicated drive/partition, and retention caps (US-4) must be enforced from day one.
- **Reolink password character bug**: special characters outside the safe set break RTSP encoding silently. Enforce the allowed set (US-2) at doorbell setup time.
- **Phase 2 RTSP divergence**: cameras behind the Home Hub use different RTSP URLs than wired cameras. Config stubs (US-7) must anticipate this, or Phase 2 onboarding will require restructuring.
