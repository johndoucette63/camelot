## Overview

Extend the F4.2 `device_annotations` pattern into a broader, first-class "notes / playbook" surface. Give the admin a place to record durable, user-curated facts about the network — quirks, schedules, conventions, contact info, playbook entries — and have the advisor chat (F4.4) pull those notes into its grounding context on every turn.

This is a deliberate counterweight to the "let the LLM decide what to remember" pattern popularized by ChatGPT's memory feature. Llama 3.1 8B is too small and too uncalibrated to be trusted with autonomous memory curation — it will save plausible-sounding nonsense, miss important things, and drift over time. In contrast, **user-curated notes are high-signal, durable, and auditable**. They become the advisor's long-term memory the same way a site runbook becomes an on-call engineer's memory: written down, reviewed, and trusted because a human decided each entry was worth keeping.

The LLM still has a role — but as a *suggester*, not a *curator*. After a conversation, the advisor can propose note additions ("You mentioned the NAS scrub happens Sunday nights — should I save that as a recurring note?") and the admin explicitly approves, edits, or rejects them. The reviewer loop is what makes this reliable.

### Why this is not "LLM auto-memory"

| Pattern | Who decides what to save | Reliability on Llama 3.1 8B |
| --- | --- | --- |
| LLM auto-memory (ChatGPT-style) | Model decides in-the-loop | Low — drifts, hallucinates, over-generalizes |
| User-curated notes | Admin writes/approves every entry | High — every entry is deliberate |
| LLM-suggested, user-approved (this feature) | Model proposes, admin approves | High — reviewer loop catches errors |

### How it fits with F4.4

F4.4's `prompt_assembler.py` already renders a grounded system prompt with device/service/alert/event sections. This feature adds a new **"Notes"** section assembled from the same live Postgres database, pulled on every turn just like the other sections. No architectural change to F4.4 — a new loader function and a new Markdown section.

## User Stories

### US-1: Per-device notes (extend F4.2 annotations)
**As a** Camelot admin
**I want** to record free-form notes about individual devices that go beyond the F4.2 `role` / `description` / `tags` fields
**So that** I can capture quirks, schedules, and context that the advisor can reference in chat

#### Acceptance Criteria

- New `device_notes` table (or extend `annotations`) supporting multiple notes per device, each with:
  - `created_at`, `updated_at`
  - `body` (Markdown text, ≤ 2 KB)
  - `pinned` boolean (pinned notes always appear in grounding context; unpinned ones may be trimmed under the context-window budget)
- Notes editable via the Devices page — click a device row → modal → add/edit/delete notes
- Notes appear on the device detail view sorted most-recent first
- At least 1 per-device note written by hand survives a backend restart and is visible in a fresh chat session

### US-2: Per-service notes
**As a** Camelot admin
**I want** to record notes about individual services (not just devices)
**So that** I can document upgrade histories, scheduled downtime, known quirks, and ownership contacts

#### Acceptance Criteria

- New `service_notes` table (or shared polymorphic notes table) keyed on `service_definitions.id`
- Notes editable from the Services page — click a service → modal → notes tab alongside the existing health history tab
- Each note supports the same Markdown / pinned / timestamp fields as per-device notes
- When the advisor chat is asked about a service with notes, the notes are visible in the prompt's grounding context

### US-3: Cross-cutting playbook entries
**As a** Camelot admin
**I want** a top-level "Playbook" / "Notes" page for entries that aren't tied to a specific device or service
**So that** I can record network-wide conventions, contact info, vendor details, and recurring maintenance schedules

#### Acceptance Criteria

- New `/playbook` route in the frontend with a simple list + editor
- Entries have: `title`, `body` (Markdown), `tags[]`, `created_at`, `updated_at`, `pinned`
- Tag-based filtering in the UI (e.g., `maintenance`, `vendor`, `convention`, `security`)
- Playbook entries are included in the advisor chat grounding context under a new "## Playbook" section
- At least 3 example seed entries in the initial deploy: NAS scrub schedule, VPN rotation schedule, DNS ownership note

### US-4: Notes in the advisor chat grounding context
**As a** Camelot admin
**I want** the advisor to automatically reference my notes when answering questions
**So that** I don't have to re-explain context every time I start a chat

#### Acceptance Criteria

- `prompt_assembler.py` gains a new `_load_notes_section` function that queries pinned per-device notes, per-service notes, and playbook entries
- New "## Notes" (or "## Playbook") Markdown section appears in the system prompt between the existing sections
- Unpinned notes are included only when total prompt size is under the character budget; under pressure, unpinned notes are the first thing trimmed (before history)
- Graceful degradation: if the notes queries fail, the section degrades to a placeholder and the rest of the prompt still assembles (same pattern as F4.4 FR-013)
- Acceptance test: seed a note "The NAS goes offline Sunday 2am for RAID scrub — not a real outage", then ask the advisor "why is the NAS offline Sunday morning?" — the response should reference the scrub, not guess

### US-5: LLM-suggested notes with user approval (reviewer loop)
**As a** Camelot admin
**I want** the advisor to proactively suggest notes I should save based on things I said during a chat
**So that** high-value context gets captured without me having to remember to write it down

#### Acceptance Criteria

- At the end of a conversation (or on explicit "suggest notes" action), the advisor issues a single extra LLM call with a focused prompt: "Based on this conversation, are there 0–3 facts about the user's network that would be worth saving as durable notes? Respond as a JSON list of {target: device|service|playbook, target_id: ..., body: ...} objects. Return [] if nothing is worth saving."
- Suggestions appear as a review panel in the UI: each entry shows the proposed target, body, and provenance (which chat turn it came from)
- Admin actions: approve (save as-is), edit (open editor pre-filled), reject (dismiss permanently)
- Nothing is persisted to the notes tables without an explicit approval click
- Suggestions that are rejected are remembered so the same suggestion isn't shown twice across conversations (small `rejected_suggestions` table, content-hashed)

### US-6: Notes visible in backups and exports
**As a** Camelot admin
**I want** notes to be part of the standard advisor database backup
**So that** my curated domain knowledge is recoverable

#### Acceptance Criteria

- Notes tables are in the same Postgres database as the rest of the advisor schema and are included in any existing DB backup/dump path
- No separate backup step required
- A fresh advisor deploy against a restored DB snapshot shows all prior notes

## Non-Goals (explicit scope exclusions)

- **Autonomous LLM memory curation**: The advisor never saves notes without explicit admin approval. No "silently learned" facts.
- **Cross-project knowledge transfer**: Notes stay in this Camelot instance. No cloud sync, no shared memory across deployments (Constitution I — Local-First).
- **Rich text / attachments**: Notes are Markdown only. No images, no file uploads, no embedded tables from Grafana. If you want a chart, link to the Grafana dashboard from inside the note body.
- **Full-text search across notes**: v1 is list + tag filter. Full-text search is a future feature if notes volume grows beyond ~50 entries.
- **Version history / undo**: Edits overwrite the previous body. A future feature could add Git-like history if needed.
- **Multi-user notes with ACLs**: Single-admin deployment. No `author_id`, no permissions.

## Dependencies

- **F4.2 (Network discovery & device inventory)** — notes attach to the existing `devices` and `device_annotations` tables.
- **F4.3 (Service registry & health dashboard)** — per-service notes attach to `service_definitions`.
- **F4.4 (AI-powered advisor chat)** — US-4 extends `prompt_assembler.py` to include the new "Notes" section; US-5 extends the chat router to trigger a suggestion-generation call.
- **F3.1 (Ollama deployment)** — US-5 requires Ollama for the suggestion-generation LLM call. Degrades gracefully if Ollama is unavailable (suggestions simply aren't generated; manual note entry still works).

## Open questions for clarify / planning

- Should per-device and per-service notes share a single polymorphic `notes` table (with `target_type` + `target_id`) or live in dedicated tables? Polymorphic is simpler to query for the advisor chat; dedicated tables are easier to FK.
- For US-5 (LLM-suggested notes), how aggressive should the trigger be? After every chat turn? Only on explicit user action? On conversation "end" (whatever that means)?
- Should pinned notes have a hard cap (e.g., max 20 pinned per category) to prevent the prompt context from ballooning?
- Does US-3's "Playbook" page deserve its own nav entry, or should it nest under the existing Devices/Services page as a tab?
