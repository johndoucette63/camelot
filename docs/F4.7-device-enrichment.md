## Overview

The current scanner runs an ARP-only ping sweep (`nmap -sn`) which discovers IPs and MACs but produces no hostname, OS, or service information for most devices. The MAC vendor lookup provides a manufacturer name (e.g., "Apple", "Raspberry Pi Foundation") but not the device type or purpose. This leaves the majority of inventory rows showing "—" for hostname and a generic vendor string, making the inventory difficult to use without manual annotation.

This feature adds a multi-protocol **enrichment pass** that runs after the existing ARP sweep. It collects device names, OS families, open services, and mDNS advertisements, then uses that data to auto-classify devices with a role and friendly name. The fast ARP sweep remains unchanged — enrichment is a separate, slower step that targets devices lacking metadata.

### Design Principles

- **Non-disruptive**: The ARP sweep loop (`scanner_entrypoint.py`) is untouched. Enrichment runs as a second pass after each scan completes, only targeting devices that need it.
- **Graceful degradation**: Each enrichment source (mDNS, nmap fingerprint, NetBIOS, SSDP) is independent. If one fails or times out, the others still contribute.
- **Incremental**: Once a device has been enriched, it is not re-scanned on every cycle. Re-enrichment is triggered only when a device's IP changes or on manual request.
- **Additive**: Enrichment data is stored in new columns. Existing columns (`hostname`, `vendor`) are only updated if the enriched value is better (non-null replacing null). User-supplied annotations always take precedence.

## User Stories

### US-1: Automatically identify unknown devices via mDNS

**As a** Camelot admin
**I want** the scanner to discover mDNS/Bonjour service advertisements on the LAN
**So that** Apple devices, Sonos speakers, printers, and smart home gear are identified by their advertised name and service type

#### Acceptance Criteria

- A background mDNS listener collects service advertisements from the `192.168.10.0/24` subnet
- Discovered mDNS names (e.g., `Johns-iPhone._companion-link._tcp.local.`) are parsed into a friendly device name
- mDNS service types are mapped to device roles (e.g., `_airplay._tcp` -> speaker, `_ipp._tcp` -> printer, `_companion-link._tcp` -> Apple device)
- mDNS results are stored on the device record and visible in the inventory
- mDNS discovery runs passively in the scanner container; no extra container required

### US-2: Fingerprint devices with nmap OS and service detection

**As a** Camelot admin
**I want** the scanner to run OS and service fingerprinting on devices that have unknown hostname or vendor
**So that** I can see what OS each device runs and what services it exposes

#### Acceptance Criteria

- After the ARP sweep, a targeted nmap scan (`-O -sV --top-ports 100`) runs against devices where `hostname IS NULL` or `os_family IS NULL`
- Detected OS family (e.g., Linux, iOS, macOS, Windows) and OS detail string are stored on the device record
- Detected open ports and service names are upserted into the existing `services` table
- Fingerprint scan is rate-limited: max 5 devices per enrichment cycle to keep scan duration reasonable
- Devices that have already been enriched are skipped unless their IP has changed since last enrichment
- nmap runs with `--host-timeout 30s` to avoid blocking on unresponsive hosts

### US-3: Discover device names via NetBIOS

**As a** Camelot admin
**I want** the scanner to query NetBIOS names on the LAN
**So that** Windows PCs and IoT devices that advertise NetBIOS names are identified

#### Acceptance Criteria

- NetBIOS name resolution runs via nmap NSE script (`nbstat.nse`) as part of the fingerprint scan
- Discovered NetBIOS names are used as the device hostname if no hostname was found via DNS or mDNS
- NetBIOS workgroup/domain is stored as metadata

### US-4: Discover devices via SSDP/UPnP

**As a** Camelot admin
**I want** the scanner to query SSDP multicast to find UPnP devices
**So that** smart TVs, media players, and IoT devices that advertise via UPnP are identified with their friendly names and model info

#### Acceptance Criteria

- An SSDP M-SEARCH multicast query is sent after the ARP sweep
- UPnP device description XML is fetched for each responder to extract: friendly name, manufacturer, model name, model number
- SSDP results are matched to existing device records by IP address
- Discovered friendly names and model info are stored on the device record
- SSDP discovery has a 10-second timeout to avoid blocking the scan loop

### US-5: Auto-classify device role from enrichment data

**As a** Camelot admin
**I want** devices to be automatically assigned a role based on their enrichment data
**So that** new devices get a reasonable classification without manual annotation

#### Acceptance Criteria

- Classification rules use a priority chain: mDNS service type > open ports > OS family > vendor string
- Example mappings:
  - mDNS `_ipp._tcp` or `_printer._tcp` -> `printer`
  - mDNS `_airplay._tcp` or `_sonos._tcp` -> `speaker`
  - mDNS `_homekit._tcp` -> `iot`
  - Open port 80/443 + OS Linux -> `server`
  - Open port 53 -> `dns`
  - OS family macOS/Windows -> `workstation`
  - OS family iOS/Android -> `iot` (mobile)
  - Vendor "Sonos" -> `speaker`
  - Vendor contains "camera" or port 554 (RTSP) -> `camera`
- Auto-classification only sets the annotation role if the current role is `unknown` or no annotation exists
- User-set roles are never overwritten by auto-classification
- Classification confidence is stored (high/medium/low) so the UI can indicate auto-classified vs. user-set

### US-6: View enrichment data in the device inventory

**As a** Camelot admin
**I want** to see enrichment metadata (OS, services, mDNS name, classification confidence) in the device table
**So that** I have a complete picture of each device at a glance

#### Acceptance Criteria

- Device table gains an "OS" column showing OS family (e.g., "Linux", "macOS", "iOS")
- Device detail view (click-through or expandable row) shows: OS detail, open ports/services, mDNS name, SSDP model, NetBIOS name, classification source
- Auto-classified roles are visually distinguished from user-set roles (e.g., italic text or a small "auto" badge)
- A "Re-scan" button on each device row triggers enrichment for that specific device on the next scan cycle

## Technical Design

### New Dependencies

| Package | Version | Purpose |
|---------|---------|---------|
| `zeroconf` | >=0.132 | mDNS/Bonjour service discovery (pure Python, async-compatible) |

nmap OS detection (`-O`) and NSE scripts (`nbstat.nse`) require no new Python packages — they use the existing `python-nmap` library. SSDP uses raw sockets (stdlib `socket` + `httpx` for XML fetch).

### Database Changes (Alembic migration `007_device_enrichment.py`)

**Extend `devices` table:**

| Column | Type | Default | Description |
|--------|------|---------|-------------|
| `os_family` | `String(50)` | `NULL` | Detected OS family (Linux, macOS, iOS, Windows, etc.) |
| `os_detail` | `String(255)` | `NULL` | Full OS string from nmap (e.g., "Linux 5.15", "Apple iOS 17.4") |
| `mdns_name` | `String(255)` | `NULL` | Friendly name from mDNS advertisement |
| `netbios_name` | `String(255)` | `NULL` | NetBIOS hostname |
| `ssdp_friendly_name` | `String(255)` | `NULL` | UPnP friendly name |
| `ssdp_model` | `String(255)` | `NULL` | UPnP model name + number |
| `classification_source` | `String(50)` | `NULL` | How role was determined: `user`, `mdns`, `nmap`, `ssdp`, `vendor`, `null` |
| `last_enriched_at` | `DateTime` | `NULL` | Timestamp of last enrichment pass |
| `enrichment_ip` | `String(15)` | `NULL` | IP address at time of last enrichment (triggers re-enrichment on change) |

No new tables required. Open ports/services continue to use the existing `services` table.

### Scanner Architecture

```
scanner_entrypoint.py  (existing loop, unchanged)
  └── run_scan()           # ARP sweep  (existing, unchanged)
  └── run_enrichment()     # NEW — called after run_scan() completes
        ├── _enrich_mdns()       # Query zeroconf cache
        ├── _enrich_nmap()       # OS + service fingerprint (max 5 devices)
        ├── _enrich_ssdp()       # SSDP M-SEARCH multicast
        └── _auto_classify()     # Apply role classification rules
```

- `run_enrichment()` lives in a new file: `app/services/enrichment.py`
- The mDNS listener (`zeroconf.Zeroconf`) is started once at scanner boot and runs continuously in a background thread, accumulating discovered services in a thread-safe dict. `_enrich_mdns()` simply reads from this cache.
- `_enrich_nmap()` runs a targeted scan on up to 5 devices per cycle: `nmap -O -sV --top-ports 100 --host-timeout 30s <ip1> <ip2> ...`
- `_enrich_ssdp()` sends a single M-SEARCH multicast and collects responses for 10 seconds.
- `_auto_classify()` runs classification rules against all enrichment data and updates annotations where role is `unknown`.

### Scanner Container Changes

The scanner container already runs with `network_mode: host` (required for ARP). This also allows mDNS multicast and SSDP multicast to work without additional configuration. nmap OS detection (`-O`) requires root — the scanner container already runs as root for ARP scanning, so no privilege changes are needed.

### API Changes

**`GET /devices`** — response gains new fields: `os_family`, `os_detail`, `mdns_name`, `netbios_name`, `ssdp_friendly_name`, `ssdp_model`, `classification_source`, `last_enriched_at`.

**`POST /devices/{mac_address}/re-enrich`** — new endpoint. Clears `last_enriched_at` so the device is picked up in the next enrichment cycle. Returns 202 Accepted.

### Frontend Changes

- Device table: add "OS" column after "Vendor" (shows `os_family`, falls back to "—")
- Device table: auto-classified roles shown with a subtle "(auto)" suffix or badge
- Device detail/expandable row: show all enrichment fields grouped under an "Identification" section
- "Re-scan" button per device row calls `POST /devices/{mac_address}/re-enrich`

## Enrichment Priority & Conflict Resolution

When multiple sources provide a hostname or name, the following priority applies (highest first):

1. **User annotation** (description field) — never overwritten
2. **DNS hostname** (from nmap reverse DNS) — existing `hostname` column
3. **mDNS name** — stored in `mdns_name`, used as display name if hostname is null
4. **NetBIOS name** — stored in `netbios_name`, fallback if above are null
5. **SSDP friendly name** — stored in `ssdp_friendly_name`, last resort

The frontend picks the best available name for display using this chain. All source values are preserved in their respective columns so nothing is lost.

## Scope Boundaries

**In scope:**
- mDNS passive listener + cache
- nmap OS/service fingerprinting on unknown devices
- NetBIOS name discovery (via nmap NSE)
- SSDP/UPnP device discovery
- Auto-classification from enrichment data
- New DB columns + migration
- API + frontend updates

**Out of scope (future work):**
- DHCP lease table integration (requires router API/SSH access — consider for F8.1)
- SNMP polling (too heavy for a home network scan)
- Active mDNS probing (passive listening is sufficient)
- Historical enrichment tracking (only latest state stored)
- ML-based device classification (rule-based is sufficient for home network scale)
