## Overview

Replace the current Torrentbox VPN topology — host-level `openvpn@pia.service` + Deluge on `docker_default` bridge — with a **VPN sidecar container** (e.g. `qmcgaw/gluetun`) that Deluge runs inside via `network_mode: service:gluetun`. This removes a class of failures surfaced by the 2026-02-23 → 2026-04-14 incident, in which a mangled shebang in `/etc/openvpn/vpn-up.sh` killed the tunnel and Deluge silently fell back to the home WAN for ~7 weeks. The current "kill-switch" is inverted (locks iptables *while* VPN is up, opens *when* it dies) — this feature replaces it with a proper default-deny model enforced inside the sidecar's network namespace.

This work is **prerequisite to F5.1 US-2** (paid private indexers): private trackers require ratio maintenance and reliable IP leak prevention, neither of which the current topology provides.

See `docs/INFRASTRUCTURE.md` (Deluge Settings → "Known architectural gaps") and the `project_vpn_incident_2026-04` memory for full incident context.

## User Stories

### US-1: Sidecar VPN container with proper kill-switch
**As a** Torrentbox operator
**I want** Deluge running inside a VPN sidecar container with a default-deny kill-switch
**So that** Deluge cannot egress traffic on any path other than the VPN tunnel — including during tunnel hiccups, container restarts, or outages

#### Acceptance Criteria
- VPN sidecar container deployed on Torrentbox (gluetun or equivalent), pinned image tag
- PIA credentials configured via container env vars or mounted secret file (NOT committed to repo)
- Deluge container reconfigured with `network_mode: service:<sidecar>`, no longer on `docker_default`
- Deluge ports exposed via the sidecar container's port mapping, not directly on the host
- Inside the sidecar netns: default-deny iptables/nftables policy, allow only VPN interface + LAN (192.168.10.0/24)
- Verification: stopping the sidecar container immediately blocks all Deluge egress (tested by `docker stop <sidecar> && docker exec deluge curl --max-time 5 ifconfig.me` returning failure, not the home WAN IP)
- Verification: `docker exec deluge curl ifconfig.me` returns a PIA exit IP under normal operation

### US-2: Tunnel-health watchdog
**As a** Torrentbox operator
**I want** an automated check that detects when Deluge's external IP is the home WAN
**So that** a silent VPN failure cannot recur for weeks without my knowledge

#### Acceptance Criteria
- Watchdog runs at least every 15 minutes (cron or systemd timer on Torrentbox, or a Grafana alert query)
- Compares `docker exec deluge curl -s --max-time 5 ifconfig.me` against the known home WAN IP (or against a list of known PIA exit ranges)
- On mismatch: alert via the existing Network Advisor alert pipeline (Phase 4.5) AND/OR stop the Deluge container automatically
- Watchdog itself is observable: a heartbeat/last-run timestamp visible in Grafana or the Advisor dashboard, so a silently-failed watchdog is also detectable
- Documented in `docs/INFRASTRUCTURE.md`

### US-3: PIA port forwarding for inbound peers
**As a** Torrentbox operator
**I want** the sidecar container to request and apply PIA's port-forwarded port to Deluge's `listen_ports`
**So that** incoming peer connections work, ratio maintenance is feasible, and private trackers (F5.1 US-2) can be onboarded successfully

#### Acceptance Criteria
- Sidecar container requests PIA port forwarding (gluetun has built-in support; `pia-foss/manual-connections` is the alternative)
- Forwarded port is automatically pinned to Deluge's `listen_ports` config (via the sidecar's port-forward callback or a small reconciliation script)
- Deluge incoming peer count is non-zero on healthy public torrents (verified after 24h of operation)
- Port re-acquisition handled gracefully on PIA token rotation (PIA forwarded ports are time-limited)

### US-4: Decommission host-level OpenVPN
**As a** Torrentbox operator
**I want** the legacy host-level `openvpn@pia.service` and its `vpn-up.sh` / `vpn-down.sh` scripts removed once the sidecar is proven
**So that** there is no chance of accidentally re-enabling the broken-by-design kill-switch model, and there is one obvious source of truth for VPN topology

#### Acceptance Criteria
- After US-1 has been stable for at least 7 days: `sudo systemctl disable --now openvpn@pia` on Torrentbox
- `/etc/openvpn/pia.conf`, `/etc/openvpn/vpn-up.sh`, `/etc/openvpn/vpn-down.sh`, `/etc/openvpn/pia-credentials.txt` archived (moved to `/etc/openvpn/legacy-014/`) rather than deleted, in case rollback is ever needed
- Host iptables OUTPUT/INPUT chains confirmed at default ACCEPT policy with zero rules (no leftover kill-switch state)
- `docs/INFRASTRUCTURE.md` updated: remove host-level VPN references, document the sidecar-only topology
- `project_vpn_incident_2026-04` memory updated to mark gaps 1, 4 closed

## Out of Scope

- Migrating Sonarr / Radarr / Prowlarr / FlareSolverr / Lidarr / LazyLibrarian — they remain on `docker_default`. Only Deluge moves behind the sidecar.
- Changing the `/mnt/nas/torrents` SMB mount or the *arr → Deluge category routing (`tv-sonarr`, `radarr`, `lidarr`).
- Switching VPN provider away from PIA. Provider stays the same; only the integration topology changes.
- Migrating to WireGuard. OpenVPN remains the protocol unless gluetun's PIA WireGuard support is materially better, in which case that's a planning-time decision.
- The F5.1 quality profiles + paid indexer work — F5.1 stays its own feature, but F5.3 must merge before F5.1 US-2 starts.

## Risks

- **Cutover window**: Deluge will be unavailable during the migration. Schedule for a low-activity window. Sonarr/Radarr will retry queued grabs on Deluge return.
- **PIA credential exposure**: gluetun reads PIA credentials from env vars or files. Use a `.env` file (gitignored) on Torrentbox, not Compose-inline values.
- **NAS mount inheritance**: Deluge needs `/mnt/nas/torrents` mounted; verify the bind-mount survives the network-mode change (it should — only network namespace changes, not the mount namespace).
- **Port-forward race**: PIA assigns the forwarded port asynchronously after tunnel establishes. Deluge may need a restart loop or a sidecar callback to pick up the port.
