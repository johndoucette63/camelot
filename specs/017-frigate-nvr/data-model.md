# Phase 1 Data Model: Frigate NVR

**Feature**: 017-frigate-nvr
**Date**: 2026-04-18

This feature touches two data surfaces: (1) the **Frigate runtime config** (not a database — YAML on disk that drives the stack) and (2) a small set of rows in the existing Advisor `alert_thresholds` table. There are **no new database tables and no schema migrations**; everything new is data, not structure.

## Advisor threshold rows (seeded by migration 009)

The existing `alert_thresholds` table (from Phase 4.5 / migration 004) has the actual shape:

```text
alert_thresholds(
  key            PRIMARY KEY,    -- string, e.g. "frigate_storage_fill_percent"
  value          NUMERIC(10,2),  -- current effective value
  unit           string,         -- "%", "ms", "s", "minutes"
  default_value  NUMERIC(10,2),
  min_value      NUMERIC(10,2),
  max_value      NUMERIC(10,2),
  updated_at     timestamp
)
```

Every threshold is one row keyed by a stable `key` string. The operator (`>=`), the rule that owns the key, and any derived semantics (e.g., "this is a sustained window") live in the rule code, not the table.

### Rows seeded by `009_frigate_thresholds.py`

| `key` | `value` | `unit` | `min` | `max` | Purpose |
|-------|---------|--------|-------|-------|---------|
| `frigate_storage_fill_percent` | `85` | `%` | `50` | `99` | FR-034/35 — warn at 85% fill on `/mnt/frigate` |
| `frigate_detection_latency_p95_ms` | `2000` | `ms` | `500` | `10000` | FR-036 — P95 detection-latency threshold |
| `frigate_detection_latency_window_s` | `300` | `s` | `60` | `3600` | FR-036/37 — sustained window (seconds) over which P95 is judged |

The latency rule reads both P95 and window from the threshold table. Three rows, not two, because the existing `alert_thresholds` schema has no `window_seconds` column — the window is stored as its own threshold row to stay schema-compatible with the existing table.

All three thresholds are **editable in-place** by the admin via the existing Advisor thresholds UI (Phase 4.5); the migration only seeds defaults.

### Relationship to existing entities

- `alert` rows generated by the two rules reuse the existing lifecycle columns (`delivery_status`, `delivery_next_attempt_at`, etc.) from migration 008.
- Notifications are delivered via the existing `notification_sinks` path, which already supports Home Assistant targets (migration 008). No new sink type.

## Identity / uniqueness

- The two rules' `rule_name` values (`frigate_storage_high`, `frigate_detection_latency`) MUST be unique and stable — they are the join key between the rule code, the threshold row, and any rule-mute row an admin later adds.
- Camera name `doorbell` is unique within `config.yml`. Phase 2 will add camera names like `solar_cam_front` following the same snake_case pattern.

## Lifecycle / state transitions

- **Privacy mode** toggles `cameras.doorbell.detect.enabled` and `cameras.doorbell.record.enabled` via MQTT `set` topics at runtime (per R10). No file rewrite; state lives in Frigate's in-memory camera state until toggled back.
- **Event clip retention**: Frigate's built-in retention loop prunes clips older than `record.events.retain.default` daily. No manual prune.
- **Continuous footage retention**: Frigate's retention loop prunes segments older than `record.retain.days`. The Advisor FR-034 rule is the safety net when these internal caps don't keep up for any reason (bug, config error, burst traffic).

## Scale assumptions

- 1 camera at ~5 Mbps main + ~500 Kbps sub = ~50 GB/day continuous. With `record.retain.days: 7` → ~350 GB baseline, comfortably within a 2 TB drive even after event-clip overhead.
- Phase 2 (3 additional solar cameras, mostly motion-activated, not 24/7) roughly doubles that ceiling — still fits 2 TB. This is the drive-sizing sanity check for R5.
